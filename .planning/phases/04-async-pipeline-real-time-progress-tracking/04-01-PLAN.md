<objective>
Build the n8n Pipeline Orchestrator workflow (14-Pipeline-Orchestrator) that replaces the frontend's sequential 3-call pattern with a single async endpoint. The webhook accepts the generation request, creates a pipeline_runs row in Supabase, returns HTTP 202 immediately with the pipeline_run_id, then continues executing all 6 pipeline steps sequentially -- updating pipeline_runs progress after each step via Supabase REST API.

Purpose: This is the backend half of real-time progress tracking. Without this orchestrator, the frontend has no pipeline_run_id to subscribe to and no real progress events flowing into the database.

Output: A deployed, ACTIVE n8n workflow on flowbound.app.n8n.cloud that orchestrates the full content generation pipeline asynchronously with per-step progress persistence.
</objective>

Build the n8n workflow JSON for "ELUXR 14-Pipeline-Orchestrator" with the following node chain:

**Node 1: Webhook** (type: n8n-nodes-base.webhook, typeVersion: 2)
- httpMethod: POST
- path: eluxr-generate-content
- responseMode: responseNode (critical -- allows custom 202 response then continues)
- options.allowedOrigins: "*"

**Node 2: Auth Validator** (type: n8n-nodes-base.executeWorkflow, typeVersion: 1)
- workflowId: S4QtfIKpvhW4mQYG (Auth Validator sub-workflow)
- Pass full input data to sub-workflow

**Node 3: Auth OK? IF** (type: n8n-nodes-base.if, typeVersion: 2)
- Condition: $json.authenticated equals true
- FALSE path -> Node 4 (401 response)
- TRUE path -> Node 5 (dedup check)

**Node 4: 401 Unauthorized** (type: n8n-nodes-base.respondToWebhook, typeVersion: 1.1)
- respondWith: json
- responseBody: { "error": "Unauthorized" }
- options.responseCode: 401

**Node 5: Check Existing Run** (type: n8n-nodes-base.httpRequest, typeVersion: 4.2)
- Check for existing running pipeline for this user
- GET to {{$env.SUPABASE_URL}}/rest/v1/pipeline_runs?user_id=eq.{{$json.user_id}}&status=eq.running&select=id
- Headers: Authorization: Bearer {{$env.SUPABASE_SERVICE_ROLE_KEY}}, apikey: {{$env.SUPABASE_SERVICE_ROLE_KEY}}
- alwaysOutputData: true (so empty results flow through)

**Node 6: Has Running? IF** (type: n8n-nodes-base.if, typeVersion: 2)
- Condition: check if response array has length > 0
- TRUE path -> Node 7 (return existing run_id as 202)
- FALSE path -> Node 8 (create new run)

**Node 7: Return Existing Run** (type: n8n-nodes-base.respondToWebhook, typeVersion: 1.1)
- respondWith: json
- responseBody: { "success": true, "status": "accepted", "pipeline_run_id": existing_id, "resumed": true }
- options.responseCode: 202

**Node 8: Create Pipeline Run** (type: n8n-nodes-base.code, typeVersion: 2)
- Prepare the INSERT body for pipeline_runs:
  ```javascript
  const userId = $input.item.json.user_id;
  const body = $input.item.json.body || {};
  return {
    json: {
      user_id: userId,
      pipeline_type: 'full_pipeline',
      status: 'running',
      current_step: 0,
      total_steps: 6,
      step_label: 'Starting...',
      step_progress: 0,
      started_at: new Date().toISOString(),
      metadata: {
        business_url: body.business_url || '',
        industry: body.industry || '',
        month: body.month || '',
        products: body.products || '',
        brand_voice: body.brand_voice || '',
        platforms: body.platforms || []
      },
      // Pass through for downstream nodes
      body: body,
      email: $input.item.json.email
    }
  };
  ```

**Node 9: INSERT pipeline_runs** (type: n8n-nodes-base.httpRequest, typeVersion: 4.2)
- POST to {{$env.SUPABASE_URL}}/rest/v1/pipeline_runs
- Headers: Authorization: Bearer {{$env.SUPABASE_SERVICE_ROLE_KEY}}, apikey: {{$env.SUPABASE_SERVICE_ROLE_KEY}}, Content-Type: application/json, Prefer: return=representation
- Body: the prepared INSERT data (user_id, pipeline_type, status, current_step, total_steps, step_label, step_progress, started_at, metadata)
- The response returns the created row with its UUID `id`

**Node 10: Extract Run ID** (type: n8n-nodes-base.code, typeVersion: 2)
- NAME THIS NODE EXACTLY "Extract Run ID" -- other nodes reference it by name.
- Extract pipeline_run_id from INSERT response and preserve body/metadata for sub-workflows:
  ```javascript
  const insertResult = $input.item.json;
  // PostgREST returns array with Prefer: return=representation
  const runId = Array.isArray(insertResult) ? insertResult[0].id : insertResult.id;
  const metadata = Array.isArray(insertResult) ? insertResult[0].metadata : insertResult.metadata;
  return {
    json: {
      pipeline_run_id: runId,
      user_id: Array.isArray(insertResult) ? insertResult[0].user_id : insertResult.user_id,
      body: metadata,
      email: $input.item.json.email || metadata.email
    }
  };
  ```

**Node 11: Respond 202** (type: n8n-nodes-base.respondToWebhook, typeVersion: 1.1)
- respondWith: json
- responseBody: `={{ JSON.stringify({ success: true, status: "accepted", pipeline_run_id: $json.pipeline_run_id }) }}`
- options.responseCode: 202
- This node has an OUTPUT that continues to the pipeline steps

**After the 202 response, the workflow continues executing:**

For each of the 6 steps, create a pair: [Update Step N] -> [Execute Sub-Workflow N]

Step labels and sub-workflow mapping:
1. "Analyzing your business..." -> 01-ICP-Analyzer
2. "Creating content themes..." -> 02-Theme-Generator
3. "Writing posts..." -> 04-Content-Studio
4. "Generating images..." -> 11-Image-Generator
5. "Creating videos..." -> 13-Video-Creator
6. "Syncing calendar..." -> 09-Calendar-Sync

**CRITICAL: pipeline_run_id recovery after Execute Sub-Workflow nodes.**

Each Execute Sub-Workflow node returns the sub-workflow's output data, which does NOT contain `pipeline_run_id`. The "Update Step N" PATCH nodes need the pipeline_run_id to build the URL `?id=eq.{{pipeline_run_id}}`.

**Solution:** In ALL "Update Step N" HTTP Request nodes AND the "Mark Complete" node, use an n8n expression that references the named "Extract Run ID" node instead of `$json.pipeline_run_id`:

```
PATCH URL: {{$env.SUPABASE_URL}}/rest/v1/pipeline_runs?id=eq.{{ $('Extract Run ID').item.json.pipeline_run_id }}
```

This n8n expression syntax `$('Node Name').item.json.field` retrieves data from a specific named node regardless of what the immediately preceding node output. This is why the "Extract Run ID" node MUST be named exactly "Extract Run ID".

Do NOT use `{{$json.pipeline_run_id}}` in any PATCH URL after the first Execute Sub-Workflow -- it will be undefined.

**For each "Update Step N" node** (type: n8n-nodes-base.httpRequest, typeVersion: 4.2):
- PATCH to `{{$env.SUPABASE_URL}}/rest/v1/pipeline_runs?id=eq.{{ $('Extract Run ID').item.json.pipeline_run_id }}`
- Headers: Authorization: Bearer {{$env.SUPABASE_SERVICE_ROLE_KEY}}, apikey: {{$env.SUPABASE_SERVICE_ROLE_KEY}}, Content-Type: application/json, Prefer: return=representation
- Body: { "current_step": N, "step_label": "<label>", "step_progress": 0, "status": "running", "updated_at": "<ISO timestamp>" }

**For each "Execute Sub-Workflow" node** (type: n8n-nodes-base.executeWorkflow, typeVersion: 1):
- workflowId: look up from the workflow files (use search_nodes or n8n MCP to find IDs)
- Wait=true (default, DO NOT use Wait=false -- known bug)
- Pass through user_id, body (business_url, industry, month, etc.), and pipeline_run_id
- Note: Sub-workflows expect specific input shapes. ICP Analyzer expects the form fields directly. Theme Generator expects session_id/icp_summary. Content Studio expects session_id/date. Image/Video/Calendar may need different data. Check the webhook body expectations of each sub-workflow and map accordingly.

**After all 6 steps complete:**

**Node "Mark Complete"** (type: n8n-nodes-base.httpRequest, typeVersion: 4.2):
- PATCH to `{{$env.SUPABASE_URL}}/rest/v1/pipeline_runs?id=eq.{{ $('Extract Run ID').item.json.pipeline_run_id }}`
- Body: { "status": "completed", "current_step": 6, "step_label": "Complete!", "step_progress": 100, "completed_at": "<ISO timestamp>", "updated_at": "<ISO timestamp>" }

**Error handling:** Add an error workflow or use n8n's "On Error" setting on the workflow to catch failures. If n8n workflow-level error handler is not straightforward, add a note in the summary that per-step error handling should be added in a gap-closure plan if needed.

- ALL PATCH URL expressions MUST use `$('Extract Run ID').item.json.pipeline_run_id` (NOT `$json.pipeline_run_id`) because Execute Sub-Workflow output replaces $json

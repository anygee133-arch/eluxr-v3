**3. Test the data flow end-to-end by tracing the shape of data at each step:**

a) Frontend sends POST to /eluxr-generate-content with body:
```json
{
  "business_url": "https://example.com",
  "industry": "Technology",
  "month": "2026-03",
  "products": "SaaS tools",
  "brand_voice": "Professional",
  "platforms": ["linkedin", "instagram"]
}
```

b) Orchestrator receives this via webhook body. Auth Validator extracts user_id. Code node prepares INSERT with metadata.

c) INSERT creates pipeline_runs row. Response returns the row with UUID id.

d) Respond 202 sends `{ success: true, status: "accepted", pipeline_run_id: "<uuid>" }`

e) Frontend receives 202, extracts pipeline_run_id, subscribes to `postgres_changes` on `pipeline_runs` filtered by `id=eq.<uuid>`

f) Orchestrator PATCHes pipeline_runs after each step (using `$('Extract Run ID').item.json.pipeline_run_id` in PATCH URL -- already set up in 04-01). Supabase Realtime pushes UPDATE event to frontend.

g) Frontend's `renderProgress()` receives the updated row and updates UI.

**4. Check for common integration issues and fix:**

- **Issue: Sub-workflow data passing.** Execute Sub-Workflow passes the ENTIRE output of the previous node as input to the sub-workflow. The sub-workflows were built to receive data via Webhook node (which puts everything in `$json.body`). When called via Execute Sub-Workflow, the data is in `$json` directly (no `.body` wrapper). Check if the sub-workflows' first nodes (after trigger) reference `$json.body` and if so, the orchestrator's Code nodes before each Execute Sub-Workflow need to wrap the data appropriately.

  Fix approach: Before each Execute Sub-Workflow call, add a Code node that shapes the data into the format the sub-workflow expects:
  ```javascript
  // For 01-ICP-Analyzer (expects webhook body fields directly):
  return {
    json: {
      body: {
        business_url: $('Extract Run ID').item.json.body.business_url,
        industry: $('Extract Run ID').item.json.body.industry,
        month: $('Extract Run ID').item.json.body.month,
        products: $('Extract Run ID').item.json.body.products,
        brand_voice: $('Extract Run ID').item.json.body.brand_voice,
        platforms: $('Extract Run ID').item.json.body.platforms
      },
      user_id: $('Extract Run ID').item.json.user_id,
      email: $('Extract Run ID').item.json.email,
      authenticated: true,
      pipeline_run_id: $('Extract Run ID').item.json.pipeline_run_id
    }
  };
  ```
  Note: Use `$('Extract Run ID')` references here too, since sub-workflow output replaces `$json`.

- **Issue: pipeline_run_id preservation -- already addressed.** 04-01 now uses `$('Extract Run ID').item.json.pipeline_run_id` in all PATCH URLs. Verify this is correctly deployed by reading the workflow via MCP. If for any reason $json references slipped through, fix them to use the named node reference.

- **Issue: SUPABASE_URL environment variable.** Verify that `SUPABASE_URL` is set as an environment variable on n8n Cloud (Settings > Variables). It should be `https://llpnwaoxisfwptxvdfed.supabase.co`. If not set, the HTTP Request nodes will fail.

- **Issue: Realtime subscription with RLS.** The frontend's supabase-js client uses the anon key and the user's JWT. The SELECT policy on pipeline_runs requires `auth.uid() = user_id`. The Realtime subscription should work because it respects RLS SELECT policies. Verify the user is authenticated before subscribing (04-02 ensures checkActivePipeline runs after auth confirmation).

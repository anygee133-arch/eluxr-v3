**TOOL-05 Fix -- Replace setTimeout with proper polling loop:**

Remove the "Code in JavaScript" node that contains:
```javascript
await new Promise(resolve => setTimeout(resolve, 35000));
return $input.all();
```

Replace with a proper Wait + IF polling loop:

**New node flow after KIE -- Create Image Task:**

1. **"Init Poll Counter"** (Code node):
```javascript
const item = $input.item.json;
return {
  json: {
    ...item,
    pollCount: 0,
    taskId: item.data?.task_id || item.task_id
  }
};
```

2. **"Wait Before Poll"** (Wait node):
- Resume: After Time Interval
- Amount: 10 (first poll) -- note: this Wait node is re-entered at 5s on retry
- Unit: Seconds
- Configure for 10s initially; the retry path can use a separate 5s Wait or the same node

**Recommended pattern (two Wait nodes for clarity):**

```
KIE -- Create Image Task
  -> Init Poll Counter (Code)
    -> Initial Wait (Wait, 10 seconds)
      -> KIE -- Get Image Result (HTTP Request, GET task status)
        -> Check Image Result (Code node)
          -> IF (isComplete)
            -> TRUE (image ready or error/timeout):
              -> IF (isSuccess)
                -> TRUE: Parse Image Result -> Image Gen Response (200)
                -> FALSE: Error Response (500 with error message)
            -> FALSE (still processing):
              -> Retry Wait (Wait, 5 seconds)
                -> loops back to KIE -- Get Image Result
```

**"Check Image Result" Code node:**
```javascript
const result = $input.item.json;
const pollCount = (result.pollCount || 0) + 1;

// KIE API returns state in data.state or similar field
const state = result.data?.state || result.state || 'processing';
const isReady = state === 'success' || state === 'completed';
const isTimeout = pollCount >= 12; // 10s + (11 * 5s) = 65s max
const isError = state === 'failed' || state === 'error';

return {
  json: {
    ...result,
    pollCount,
    isComplete: isReady || isTimeout || isError,
    isSuccess: isReady,
    error: isTimeout ? 'Image generation timed out after 65 seconds' :
           isError ? `Image generation failed: ${state}` : null
  }
};
```

**"KIE -- Get Image Result" HTTP Request node:**
- Keep the existing HTTP Request configuration from the monolith
- It polls the KIE API for the image task status using the task_id

**Loop wiring:**
- The IF node's FALSE output connects to the "Retry Wait" (5 seconds) node
- "Retry Wait" connects back to "KIE -- Get Image Result"
- This creates a loop that polls every 5 seconds until success, error, or timeout (12 attempts)

**Important:** When connecting the loop back, ensure the data flows correctly. The "Check Image Result" Code node outputs the updated pollCount, which the HTTP Request node passes through.

**TOOL-06 Fix -- Swap Video Ready? IF node wiring:**

The monolith has the connections INVERTED:
- TRUE (successFlag=1, video IS ready) -> Video Processing -- Response (WRONG)
- FALSE (successFlag!=1, video NOT ready) -> Parse Video Result (WRONG)

**Correct wiring:**
- TRUE (output 0, successFlag=1) -> Parse Video Result -> Video Gen -- Response (200 with video data)
- FALSE (output 1, successFlag!=1) -> Video Processing -- Response (200 with "still processing" message)

In the sub-workflow JSON, set the connections for "Video Ready?" correctly:
```json
"Video Ready?": {
  "main": [
    [{ "node": "Parse Video Result", "type": "main", "index": 0 }],
    [{ "node": "Video Processing — Response", "type": "main", "index": 0 }]
  ]
}
```

Compare with the correctly-wired "Image Ready?" node from the monolith as a reference pattern.

**Also consider adding a polling loop** similar to the image generator (TOOL-05 fix). The current video workflow has a single Wait + single poll. If the video is not ready after the first check, it returns "processing" -- the frontend then needs to poll again. This could be improved with a retry loop, but it is acceptable as-is since the frontend already handles the retry. Decide based on whether the existing pattern works well enough:

- Option A (minimal): Keep single Wait + single poll, return processing if not ready (frontend retries)
- Option B (improved): Add polling loop like image generator (more robust, fewer frontend round-trips)

Choose based on the existing frontend behavior and the KIE video generation typical time (videos take much longer than images, potentially minutes). If videos take >60 seconds, a single server-side poll with frontend retry is more appropriate than a long-running server-side loop.

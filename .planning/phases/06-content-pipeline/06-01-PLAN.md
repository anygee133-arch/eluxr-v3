<task type="auto">
  <name>Task 2: Restructure Pipeline Orchestrator to webhook callback pattern</name>
  <files>workflows/14-pipeline-orchestrator.json</files>
  <action>
Restructure the Pipeline Orchestrator workflow (14-pipeline-orchestrator.json) from synchronous sub-workflow chaining to a webhook callback pattern. This is CRITICAL to avoid n8n Cloud Starter plan's 5-minute execution timeout.

**Current pattern (BROKEN for real workloads):**
Orchestrator receives request -> Auth -> Dedup -> Respond 202 -> sequentially calls 6 sub-workflows via HTTP Request (each WAITS for completion) -> Mark Complete. Total: 10-20+ minutes = TIMEOUT.

**New pattern (each stage independent):**

A. **Keep existing entry flow:** Webhook (POST eluxr-generate-content) -> Auth Validator -> Auth OK? -> 401 on failure -> Check Existing Run -> Has Running? -> Return Existing Run.

B. **Modify the "no existing run" path:**
   1. Extract Run ID (keep existing Code node that creates pipeline_run in Supabase)
   2. Respond 202 with pipeline_run_id (keep existing)
   3. Update Step 1 in Supabase (PATCH pipeline_runs: current_step=1, step_label="Analyzing your business...")
   4. Fire-and-forget HTTP POST to `01-ICP-Analyzer` webhook (`eluxr-phase1-analyzer`) with body: `{ user_id, business_url, industry, month, platforms, pipeline_run_id, callback_url }`. Set `neverError: true` so orchestrator doesn't fail if sub-workflow takes long. The callback_url should be the orchestrator's callback webhook URL.
   5. **Execution ends here** (no more nodes after the fire-and-forget). Total time: < 30 seconds.

C. **Add a NEW callback webhook** (second trigger in same workflow, path: `eluxr-pipeline-callback`, POST method):
   1. Webhook receives: `{ pipeline_run_id, step, status, data }`
   2. Code node "Route Next Stage" determines what to fire next based on completed step:
      - Step 1 complete (ICP) -> Update step 2, fire 02-Theme-Generator
      - Step 2 complete (Themes) -> Update step 3, fire 04-Content-Studio with week_number=1
      - Step 3 complete (Week 1) -> Update step 4, fire 04-Content-Studio with week_number=2
      - Step 4 complete (Week 2) -> Update step 5, fire 04-Content-Studio with week_number=3
      - Step 5 complete (Week 3) -> Update step 6, fire 04-Content-Studio with week_number=4
      - Step 6 complete (Week 4) -> Mark pipeline complete (PATCH status=completed, completed_at=NOW())
   3. "Update Progress" HTTP Request node: PATCH pipeline_runs with next step info
   4. "Fire Next Stage" HTTP Request node: POST to next sub-workflow webhook with fire-and-forget
   5. IF step 6 -> "Mark Complete" branch (PATCH pipeline_runs status=completed)

D. **Pipeline step mapping (6 total steps, matching frontend progress):**
   - Step 1: "Analyzing your business..." -> fires 01-ICP-Analyzer
   - Step 2: "Creating content themes..." -> fires 02-Theme-Generator
   - Step 3: "Generating Week 1 content..." -> fires 04-Content-Studio (week=1)
   - Step 4: "Generating Week 2 content..." -> fires 04-Content-Studio (week=2)
   - Step 5: "Generating Week 3 content..." -> fires 04-Content-Studio (week=3)
   - Step 6: "Generating Week 4 content..." -> fires 04-Content-Studio (week=4)

E. **Each sub-workflow will receive and use:**
   - `pipeline_run_id` (to identify the run)
   - `callback_url` (the orchestrator callback webhook URL, e.g., `https://flowbound.app.n8n.cloud/webhook/eluxr-pipeline-callback`)
   - `user_id` (for Supabase queries)
   - `business_url`, `industry`, `month`, `platforms` (context data)
   - `week_number` (for Content Studio, 1-4)

F. **Auth on callback:** The callback webhook does NOT need Auth Validator (it's called by other n8n workflows, not the browser). Use a simple shared secret check: callback requests must include a header `X-Pipeline-Secret` with a hardcoded value (e.g., "eluxr-pipeline-internal-2024") to prevent external abuse.

G. **Error handling:** If any sub-workflow fails, it should still call the callback with `status: "error"` and `error_message`. The callback handler should PATCH pipeline_runs with status=failed and error_message.

H. **Forward the user's Authorization header** to sub-workflow calls so Auth Validator in each sub-workflow can validate the user. Store the original auth token in the pipeline_run creation step and include it in each callback-triggered fire-next call.

**n8n JSON structure notes:**
- Use the same node ID prefix pattern (orch-0001-...) but extend with new IDs
- Both webhooks (entry + callback) can coexist in the same workflow -- n8n supports multiple triggers
- Use `alwaysOutputData: true` on HTTP Request nodes that might get empty responses
- Fire-and-forget: HTTP Request with `neverError: true` and no downstream dependency on the response
